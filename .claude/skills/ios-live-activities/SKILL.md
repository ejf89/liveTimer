# iOS Live Activities in Expo/React Native

How to build and bridge a Live Activity in _this_ project. Patterns verified against the
Expo SDK 56 Modules API and Apple's ActivityKit docs. Read alongside the invariants in
`CLAUDE.md` — this skill shows the correct API shapes; CLAUDE.md says which behaviors are
non-negotiable.

## Mental model

A Live Activity has three cooperating pieces:

1. **`ActivityAttributes`** — the data contract. Static fields (fixed for the session) +
   a nested `ContentState` (the changing data). Lives in `targets/widget/` and a synced
   copy in `modules/study-timer/ios/`. ActivityKit matches an activity to its widget by the
   attributes **type name + Codable shape**, so two compiled copies in different modules
   interoperate. Keep them byte-identical.
2. **The widget extension** (SwiftUI) — renders the lock screen + Dynamic Island from
   `context.attributes` and `context.state`. Generated by `@bacons/apple-targets`.
3. **The native bridge** (Expo Module, Swift) — calls `Activity.request/update/end` and
   exposes a typed async API to JS.

## ActivityKit lifecycle (the bridge side)

All call sites are `@available(iOS 16.2, *)` with a runtime `if #available` check; older OS
and Android resolve to no-ops.

```swift
import ActivityKit

// Gate first — respects user/system setting.
guard ActivityAuthorizationInfo().areActivitiesEnabled else { throw ... }

// START
let attributes = StudyAttributes(sessionId: id, sessionName: name, goalSeconds: goal)
let content = ActivityContent(
    state: StudyAttributes.ContentState(startAnchor: anchor, isPaused: false, pausedElapsed: 0),
    staleDate: nil                          // we tick on-device; no server staleness
)
let activity = try await Activity.request(
    attributes: attributes,
    content: content,
    pushType: nil                           // nil = local updates only (no APNs)
)

// UPDATE (only on real state changes — pause/resume/name/goal; never per second)
await activity.update(
    ActivityContent(state: newState, staleDate: nil)
)

// END
await activity.end(
    ActivityContent(state: finalState, staleDate: nil),
    dismissalPolicy: .immediate             // leave the lock screen promptly
)

// ENUMERATE (zombie cleanup / reconciliation on launch)
for activity in Activity<StudyAttributes>.activities {
    await activity.end(nil, dismissalPolicy: .immediate)
}
```

Key facts:

- `Activity.request` is `throws` + `async`. `update`/`end` are `async` (not throwing).
- `pushType: nil` for local-only. `.token` is for APNs push updates (scale; out of scope).
- `dismissalPolicy`: `.immediate`, `.default`, or `.after(Date)`.
- Serialize lifecycle on the **main actor** and `await` the `end` before the next `start`,
  or rapid start/stop orphans activities (zombie rule).

## The non-negotiable timer pattern

**Never** push an update every second. Render the running timer natively so iOS animates it
on-device (works backgrounded/killed, zero bridge traffic):

```swift
// Running: count up from the anchor.
Text(timerInterval: context.state.startAnchor...Date.distantFuture, countsDown: false)
    .monospacedDigit()

// Paused: show the frozen value (no live ticking).
Text(formatHHMMSS(context.state.pausedElapsed))
```

Anchor math (JS owns it; see CLAUDE.md):

- Running → widget ticks from `startAnchor`.
- Pause → `pausedElapsed = now − startAnchor`; show frozen `pausedElapsed` + "Paused".
- Resume → `startAnchor = now − pausedElapsed`; ticking resumes at the right value.

## SwiftUI widget shape

```swift
@available(iOS 16.2, *)
struct StudyLiveActivity: Widget {
  var body: some WidgetConfiguration {
    ActivityConfiguration(for: StudyAttributes.self) { context in
      // Lock screen / banner
    } dynamicIsland: { context in
      DynamicIsland {
        DynamicIslandExpandedRegion(.leading)  { /* name */ }
        DynamicIslandExpandedRegion(.trailing) { /* time */ }
        DynamicIslandExpandedRegion(.center)   { /* progress ring */ }
      } compactLeading: { /* icon */ }
        compactTrailing: { /* time */ }
        minimal: { /* time */ }
    }
  }
}
```

- **Progress ring:** prefer `ProgressView(timerInterval:countsDown:)` for native fill animation.
  Verify it actually animates the fill in this iOS version — if it stalls, fall back to a linear
  `ProgressView(value:)` / lock-screen bar (the spec mockup uses a bar anyway). Don't let a
  finicky ring block the milestone.
- The `@main WidgetBundle` lists the widget; gate with `if #available(iOS 16.2, *)`.

## Expo Modules API (the bridge, Swift)

```swift
import ExpoModulesCore
import ActivityKit

public class StudyTimerModule: Module {
  public func definition() -> ModuleDefinition {
    Name("StudyTimer")

    // Synchronous getter
    Function("areEnabled") { () -> Bool in
      if #available(iOS 16.2, *) { return ActivityAuthorizationInfo().areActivitiesEnabled }
      return false
    }

    // Typed args via Record; async work returns a Promise implicitly
    AsyncFunction("start") { (opts: StartOptions) -> String in
      // ... Activity.request ...
      return activity.id
    }.runOnQueue(.main)   // ActivityKit lifecycle on the main actor

    AsyncFunction("end") { (id: String) in /* ... */ }.runOnQueue(.main)
    AsyncFunction("endAll") { () in /* ... */ }.runOnQueue(.main)
    AsyncFunction("getActiveSessions") { () -> [[String: Any]] in /* ... */ }
  }
}

struct StartOptions: Record {
  @Field var id: String
  @Field var name: String
  @Field var startAnchor: Double      // epoch seconds; convert to Date
  @Field var goalSeconds: Double?
}
```

- `Function` = sync (≤8 args). `AsyncFunction` = async; return value auto-wraps to a resolved
  JS promise, thrown errors auto-reject.
- Use `Record` + `@Field` for typed object args (not loose dictionaries).
- Dates cross the bridge as epoch numbers — convert at the boundary.
- `.runOnQueue(.main)` for ActivityKit calls.

## Common failure modes (check these first)

- **Build/Swift compiler weirdness in the widget:** RN ships uncompiled, raising widget build
  complexity. Clear SwiftUI preview cache: `xcrun simctl --set previews delete all`.
- **Edited `ios/` and it vanished:** `ios/` is generated. Edit `modules/` + `targets/` source
  and the plugin config, then `npx expo prebuild -p ios --clean`.
- **Activity starts but widget never renders:** attributes type drift between the two copies
  (name or Codable shape differs), or the widget bundle didn't include the activity.
- **Three copies of `StudyAttributes`:** `_shared/` links to app target _and_ widget. If no
  app-target Swift imports it, move it to plain `targets/widget/` membership → exactly two
  copies (widget + module pod).
- **Zombies after rapid start/stop:** not awaiting `end` before the next `start`, not guarding
  `start()` against re-entrancy, or not sweeping `endAll()` on stop. Verify with paired
  start/end lines in os_log; 0 activities must remain after a stop.
- **Live Activity won't appear:** missing `NSSupportsLiveActivities` (main app Info.plist) or
  user disabled Live Activities (`areActivitiesEnabled` is false).

## Verify like the rubric

After any change: build clean on the sim, then check by hand — start appears in ~1–2s, timer
advances with no per-second pushes, pause freezes + "Paused", resume continues, stop disappears,
background/kill keeps ticking, rapid start/stop leaves 0 activities, Dynamic Island shows all
three presentations.
