Live Timer — Architecture Walkthrough

A React Native / Expo study timer whose state is mirrored to a native iOS Live Activity (lock screen + Dynamic Island). React Native owns the timer; a hand-written native bridge drives ActivityKit; a SwiftUI widget renders it. This page explains how the pieces fit, exactly what data crosses the boundary, and why the load-bearing decisions were made.

JavaScript / TypeScript Swift Apple framework (ActivityKit / SwiftUI)

1 Four compilation units, one timer

The system is split across four separately-compiled units. Almost every design decision falls out of that split — they can only talk through narrow, typed seams.

JS / TS React Native app

App.tsx · hooks/useTimer.ts · lib/timer.ts · lib/format.ts

Owns the timer state machine (idle | running | paused | completed) and all the time math. A 250 ms interval drives only the in-app HH:MM:SS readout — it never touches the Live Activity. State changes call the bridge.

▼   typed async calls  start / update / end / endAll / getActiveSessions  (dates as epoch seconds)

TS Bridge — JS facade

modules/study-timer/index.ts · src/StudyTimer.types.ts · index.android.ts / .web.ts

The clean typed surface the app imports. On iOS it resolves to the native module via requireNativeModule; on Android/web Metro picks the no-op stub so shared JS never crashes.

▼   Expo Modules API (the actual JS↔Swift boundary)

Swift Bridge — native module (a CocoaPod)

modules/study-timer/ios/StudyTimerModule.swift · StudyAttributes.swift (copy)

Translates typed calls into ActivityKit lifecycle ops: Activity.request / .update / .end. Converts epoch-seconds → Date at the boundary. Every call site is @available(iOS 16.2, *). This is the only place that talks to ActivityKit.

▼   ActivityKit   matches activity ⇄ widget by type name + Codable shape

Swift Widget extension SwiftUI

targets/widget/StudyLiveActivity.swift · StudyWidgetBundle.swift · _shared/StudyAttributes.swift · _shared/TimerControlIntent.swift · _shared/TimerMath.swift

A separate process. Renders the lock-screen view + Dynamic Island (compact / expanded / minimal). The running clock is drawn with SwiftUI's Text(timerInterval:), so iOS ticks the seconds on-device — even when the app is backgrounded or killed.

Key insight: the bridge pod and the widget extension are different binaries that never share code at runtime. ActivityKit links a live Activity to its widget purely by the StudyAttributes type name and its Codable shape — which is why the attributes struct has to exist, with an identical Codable shape, in both. A drift-guard test fails if the two copies diverge (see §4).

2 What actually crosses the bridge

ActivityKit splits an activity's data in two: Attributes are fixed for the activity's life; ContentState is the part you update. Getting the right field on the right side is the whole game.

StudyAttributes static · set once at start

Passed to Activity.request(...). Immutable for the session.
FieldTypeMeaning
sessionIdStringCaller-owned id (RN session)
sessionNameStringDisplay name, e.g. "Chapter 5"
goalSecondsDouble?Optional target → progress ring/bar

ContentState dynamic · pushed on change

Sent on start and every update. Never per-second.
FieldTypeMeaning
startAnchorDatenow − elapsed; the clock counts up from here
isPausedBoolRunning vs frozen presentation
pausedElapsedDoubleFrozen seconds, shown while paused

Bridge API surface the typed seam (TS → Swift)

MethodArgsReturnsPurpose
areEnabled()—booleanLive Activities allowed? (sync getter)
start(opts)id, name, startAnchor, goalSecondsPromise<string>Begin activity; returns the ActivityKit id
update(opts)id, isPaused, startAnchor, pausedElapsedPromise<void>Push a real state change (pause/resume)
end(id)idPromise<void>End one activity
endAll()—Promise<void>Zombie sweep — end every stray
getActiveSessions()—Promise<ActiveSession[]>Full state, for relaunch reconciliation

The id gotcha: start() returns the ActivityKit-assigned id, not the RN session id. update() / end() match on that returned id. (An early bug tracked the RN id instead, so Stop updated the UI but leaked the activity — it kept ticking on the lock screen.)

3 Why the timer ticks without messages — and the anchor math

The single most important decision: the widget is never told the current time. It's told a startAnchor once, and SwiftUI counts up from it locally. Pause/resume is just arithmetic on that anchor.

in-app display & widget both derive:  elapsed = now − startAnchor

RUNNING
Widget renders Text(timerInterval: startAnchor…)
iOS animates the seconds on-device. Zero JS↔native traffic while running — works backgrounded & after the app is killed.
PAUSE
Freeze the clock
pausedElapsed = now − startAnchor  ·  isPaused = true  →  widget shows the frozen value + "Paused". One update() call.
RESUME
Re-anchor so it continues seamlessly
startAnchor = now − pausedElapsed  ·  isPaused = false  →  the on-device clock picks up exactly where it froze. One update() call.
GOAL
Stop at the finish line
When elapsed ≥ goal the session completes and the clock halts at the goal value. The widget freezes itself on-device with Text(timerInterval: … , pauseTime: startAnchor + goal) — so it stops at the goal even backgrounded or killed — and JS pushes one completion update() so it shows "Goal reached". Completion is derived (isPaused && pausedElapsed ≥ goal), so the data contract is unchanged.

Contrast: the naive approach is a setInterval pushing the new time to the widget every second. That's a flood of cross-process messages, drains battery, and stops the moment the app is suspended — the opposite of what a lock-screen timer needs. Anchoring makes the common case (running) cost nothing.

4 Load-bearing decisions & their "why"

Each of these looks like a small detail but changes how the feature behaves.

On-device ticking, not per-second pushes

Text(timerInterval:) lets iOS draw the running clock itself.

Why: real-time updates, background/killed survival, and near-zero battery — all for free. The bridge only fires on genuine state changes.

Time from a startAnchor, never a stored counter

One epoch-seconds anchor is the source of truth for both the JS display and the native render.

Why: no drift between two clocks. The pure math is unit-tested in lib/timer.ts; its one native counterpart lives in _shared/TimerMath.swift — used by both the widget and the App Intent — so there's a single Swift copy, not three.

The goal stops the timer, on-device

At the goal the session completes and the clock freezes at the goal value — via the widget's pauseTime, not a JS timer.

Why: a lock-screen timer must stop at the goal even with the app backgrounded or killed. JS adds one completion update() for the "goal reached" look; completion is derived from existing state, so the contract is unchanged.

Two copies of StudyAttributes

One in the widget (_shared/), one in the bridge pod — same Codable shape, synced by hand and enforced by a drift-guard test.

Why: ActivityKit matches by type name + Codable shape across separate binaries. A Jest guard (lib/attributesSync.test.ts) fails if the copies diverge; at scale, a shared local Swift Package would remove the duplication entirely.

Interactive intent lives in _shared/

TimerControlIntent compiles into both the widget and the app target.

Why: the widget builds the Button(intent:); the app is where perform() actually runs (launched in the background, no UI). It mutates the activity directly in Swift.

No zombies — one activity, ever

endAll() on start collapses any strays to a single activity; end(id) on stop removes the tracked one.

Why: rapid start/stop must never orphan a Live Activity. The bridge guards start() against re-entrancy and stop() sweeps endAll(); verified 8× rapid start/stop → 0 activities (paired start/end in os_log). An actor-isolated controller is the native defense-in-depth version if other callers ever appear.

Killed = persists; relaunch reconciles

The activity keeps ticking after an app kill (it's free). On launch & on foreground, the app reads getActiveSessions() and adopts or clears.

Why: better demo + it's the moment lock-screen control taps (which mutate the activity in Swift) get mirrored back into the app.
